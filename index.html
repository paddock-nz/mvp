MVP definition

What we are building — and what we are not.

The MVP is a single-farm Compliance Agent covering Canterbury's N-cap requirements. It must be able to: accept nitrogen application inputs, maintain a running kg N/ha balance, flag breach risk with an 8-week forward projection, and generate a basic PDF report a farmer could show to an auditor. That is the entire scope of week 8. FarmAdvisor Agent, multi-region rule sets, and Overseer integration are explicitly post-MVP.

In scope — MVP

// data input

Manual N application logging

Farmer enters date, paddock, product name, and kg applied via a simple web form or SMS command. Product-to-N conversion handled automatically using a built-in product database.

// rule engine

Canterbury N-cap rule set only

Single regional council config file. Checks running total against 190 kg N/ha limit. Season resets 1 June each year. Paddock-level and farm-level views.

// risk model

Breach probability & projection

Given current rate of application, projects kg N/ha at 31 July. Calculates headroom remaining. Triggers alert SMS when projected total exceeds 160 kg/ha (8-week buffer).

// output

Basic N-cap PDF report

LLM-generated report showing season-to-date application per paddock, projected year-end total, and risk status. Formatted to match what a Canterbury FEP auditor expects to see.

// notification

SMS alert on breach risk

Single SMS sent to farmer when risk score exceeds threshold. Message written in plain English, includes specific paddock recommendations to reduce N before deadline.

// pilot

One Canterbury farm, end of week 8

Minimum viable demo: a real farmer can log an application, see their running balance, and receive an alert. This is the signal that justifies the next funding conversation.

// explicitly out of scope — MVP

Overseer API integration · Ravensdown/Ballance fertiliser import · Multi-region rule sets (Waikato, Southland) · FarmAdvisor Agent · FWFP document generation · Advisor white-label portal · Mobile app · Multi-farm accounts · Fonterra supplier integration

CA-002
Architect
Design and implement the core data schema
Define the database schema for the core entities: Farm, Paddock, NApplication, SeasonBalance, and AuditEvent. Every N application logged must be immutable — updates are new entries, not edits. The AuditEvent table is append-only and cryptographically timestamped from day one, even in MVP. Design the schema to accommodate multi-region rules in future without breaking changes. Produce an ERD and share with the team before writing a line of code.

CA-003
Junior
Build the nitrogen product database and N% conversion library
Create a Python library that takes a fertiliser product name and weight applied and returns kilograms of elemental nitrogen. Seed an initial database of the 40 most common NZ dairy fertiliser products — urea (46% N), DAP (18% N), CAN (27% N), SustaiN, Flexi-N, and others sourced from Ravensdown and Ballance product sheets. Include a fuzzy-match function so farmers can type "urea" or "Urea 46" and get the right result. This library will be imported by the rule engine in Sprint 2, so the interface matters — agree it with the Architect before building.

Sprint 2 · Weeks 3–4
Intelligence — rule engine & breach model
Architect Junior Shared
7 stories
Goal: breach risk score and 8-week projection running live

Sprint 2 is where the product becomes intelligent. The Architect builds the rule engine that evaluates a farm's running balance against the Canterbury config and computes a risk score. Junior builds the breach probability model — the formula that projects where the farm will end up at 31 July if the current application rate continues. You test both against the pilot farm's real data, write the alert copy, and define the threshold logic. By end of week 4, the system can tell a farmer whether they are on track, at risk, or in breach — with a specific projected date if they are.

CA-008
Architect
Build the rule engine core — config-driven compliance evaluation
Implement a rule engine that reads the Canterbury config file (CA-004) and evaluates a farm's SeasonBalance against it. The engine must return: current kg N/ha per paddock, current kg N/ha for the whole farm, percentage of cap used, and a simple COMPLIANT / AT_RISK / BREACH status. The engine must be purely config-driven — no hardcoded Canterbury-specific logic anywhere. A new regional council should be addable by dropping in a new YAML config with zero code changes. Write comprehensive unit tests against the Canterbury config, including edge cases: paddocks of different sizes, mid-season consent changes, and applications logged in a previous season.
rule-enginecoreweek 3
CA-009
Junior
Implement the breach probability and projection model
Build a Python module that takes a farm's current season balance and projects forward to 31 July. Inputs: applications to date, days elapsed in season, days remaining to 31 July, and the farm's 3-year average application rate (or regional average if no history). Output: projected kg N/ha at season end, headroom remaining, risk score (0–1), and the specific date the farm would breach if the current rate continues. Formula: risk_score = (current_rate_kg_per_day × days_remaining) / headroom_remaining. This module is called by the rule engine after every application event and also on a nightly scheduled run. Write tests against known scenarios — a farm at 50% cap in March should score low; a farm at 85% cap in May should score high.
pythonmlweek 3
CA-010
Architect
Build the farm dashboard API — balance, projection & status
Implement GET /farms/{id}/dashboard which returns everything the farmer UI needs in one call: per-paddock N balance table, whole-farm balance, risk score, projected total at 31 July, headroom remaining in kg, headroom remaining as percentage, status flag, and the projected breach date if applicable. This endpoint is called on every page load and after every application is logged. Optimise for latency — a farmer on a slow rural connection should see a result in under 2 seconds. Cache the nightly projection and recompute in real-time only when a new application is logged.
apibackendweek 3–4
CA-011
Junior
Build the SMS alert pipeline using Twilio
Implement the SMS notification system using Twilio (NZ number). The alert triggers when a farm's risk score exceeds 0.65. A single SMS is sent per threshold crossing — not on every application. Message content is written by the Founder (CA-012) and templated here. The pipeline must be idempotent — if the nightly job runs twice, the farmer gets one SMS, not two. Log every SMS sent to the audit trail with timestamp and message body. Implement a simple opt-out mechanism (reply STOP) that disables further alerts for that farm. Test on a real NZ number before Sprint 2 ends.

CA-012
You
Write all farmer-facing alert copy and define threshold logic
Write the SMS message templates for each alert level. There are three: an early heads-up at 65% risk score, a serious warning at 85%, and a critical alert at 95%. Each message must be under 160 characters, name the specific farm and current N balance, give one concrete action the farmer can take, and not sound like a robot. Also define the threshold values and cooldown periods (how many days between repeat alerts) — these become config, not hardcoded. Test draft messages on the pilot farmer before they go live. If the farmer finds them confusing or alarming in the wrong way, rewrite them.

Architect
Implement nightly scheduled job — balance refresh & alert evaluation

Sprint 3 · Weeks 5–6
Delivery — LLM report, gap detection & audit trail
Junior Architect You
7 stories
Goal: generate a PDF report a real auditor would accept

Sprint 3 is where the product earns its credibility. The LLM report generator turns the audit trail into a PDF document a Canterbury auditor would accept. Junior owns the Anthropic API integration and prompt engineering — this is the deepest AI work in the MVP and where their learning curve is steepest. CA-015
Junior
Integrate Anthropic API and build the report generation prompt
Integrate the Anthropic Claude API (claude-sonnet-4-6) into the backend. Build the report generation prompt that takes structured audit trail data — paddock-by-paddock N balance, season dates, farm name and location, risk score, and the Canterbury rule set parameters — and produces a well-structured compliance narrative. The LLM receives only structured data, never raw sensor inputs or free-form farmer text. The prompt must instruct the model to: write in plain English a Canterbury FEP auditor expects, include all required data fields, flag any gaps or inconsistencies in the data, and never fabricate numbers. Iterate on the prompt until the output reads like a document a rural advisor would be proud to submit. This story requires multiple prompt iterations — budget the full 4hr daily focus for the week.

Architect
Build the PDF generation pipeline
Implement a PDF generation pipeline that takes the LLM's text output and produces a formatted PDF report. validationstakeholderweek 5
CA-018
Junior
Build the gap detection model
Implement a Python module that analyses a farm's N application history for suspicious gaps and generates farmer prompts. Three gap types to detect: (1) a paddock with zero applications for 6+ weeks during peak season (July–February), (2) a whole-farm N total that is more than 40% below the regional average for that soil type and stocking rate, and (3) a streak of identical application entries that may indicate copy-paste data entry rather than real records. Gaps do not trigger alerts — they trigger prompts, surfaced in the farmer dashboard as a gentle question: "We haven't seen any applications for Block 3 this season — is that right?" The module runs as part of the nightly job (CA-014). Use S-map soil data for regional average benchmarks.
pythondata-qualityweek 6
CA-019
Architect
Harden the audit trail to legal-grade standard
The audit trail written in Sprint 1 needs to become legally robust before the product goes in front of a pilot farm. Implement: cryptographic timestamping of every event using a hash chain (each event hashes the previous event's hash), a read-only API endpoint for audit trail export, tamper detection that alerts the team if any past event's hash doesn't verify, and a mechanism for the farmer or their advisor to access and download their complete audit history at any time. Document the audit trail spec clearly enough that a lawyer could understand what it is and why it is trustworthy. This is not gold-plating — it is what makes the report defensible in a resource consent dispute.

Sprint 4 · Weeks 7–8
Pilot ready — hardening, live farm & investor demo
You Architect Junior
7 stories
Goal: live Canterbury farm + investor-ready demo

Sprint 4 is about making the product real and making it show. The pilot farm goes live — real applications logged, real alerts fired, real report downloaded. Junior handles on-the-ground support and any issues that surface in live use. The Architect focuses on stability, error handling, and basic security hardening. You run the pilot with the farmer, document the outcomes, and build the investor demo narrative. By end of week 8, you have a live product, a real farmer who has used it, and a demo you can show to DairyNZ and AgriZeroNZ.

CA-022
Architect
Security hardening and basic authentication
Implement minimum viable security before a real farmer's data goes into the system. Required: JWT-based authentication for all API endpoints, HTTPS enforced everywhere, basic rate limiting on the input API to prevent accidental data floods, environment secrets rotated and stored in AWS Secrets Manager (not .env files in the repo), and a security review of the database — no public endpoints, RDS in private subnet, connection pooling correctly configured. Also implement basic error logging and alerting so the team knows immediately if something breaks in production. This is not a full security audit — it is the minimum required to responsibly hold a farmer's compliance records.

CA-023
Junior
On-ground pilot farm setup and live data entry
Junior visits or video-calls the pilot farm in week 7 to walk through the system together. Document any confusion or friction — every point of friction is a product bug, not a user error. If the farmer's records don't match the system's calculation, find out why before leaving the session. This is the most important real-world validation in the MVP.
piloton-groundweek 7
CA-024
Architect
End-to-end testing and error handling hardening
Write and run a full end-to-end test suite covering the critical user journey: farm onboarded → application logged → balance updated → risk score computed → alert SMS sent → PDF report generated → audit trail verified. All happy-path tests must pass.
